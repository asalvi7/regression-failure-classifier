## Table 1: TC_MASTER

**Purpose:** Master registry of all test cases. Tells us what each test case *is* — its identity, module, and context. Has no execution or failure information.

**Row count:** 33,890

### Column Definitions

| Column | Type | Nullable | Description |
|---|---|---|---|
| `TC_ID` | Integer | No | **Primary key.** Unique numeric ID for each test case |
| `JIRA_ID` | String | No | Jira ticket ID (e.g. `IAPP-291777`) |
| `MODULE` | String | No | Product module the test belongs to |
| `J_COMPONENT` | String | No | Specific component/feature area within the module |
| `AUTOMATED_TC_ID` | String | Yes | Name of the automation script. **One script can map to multiple TC_IDs** — do not confuse with TC_ID |
| `AUTOMATED_BY_USERID` | String | Yes | Engineer who wrote the automation script |
| `FUNC_AREA` | String | Yes | Functional area (e.g. Maintenance, Report, Invoicing) |
| `BASELINE_TIME` | Integer | Yes | Baseline execution time |
| `J_LABEL` | String | Yes | Jira labels as a JSON array (e.g. `["smoke-test", "ONA"]`) |
| `FIXED_VERSION` | String | Yes | Version in which the test was introduced or fixed |
| `CONTINENT` | String | Yes | Geography scope of the test |
| `MAINTENANCE_DONE_BY` | String | Yes | Engineer who last maintained the script |
| `DEPENDENCY` | String | Yes | Dependencies on other test cases |
| `SUB_COMPONENT` | String | Yes | Sub-component classification |

## Table 2: HIS_EXEC_REPORT

**Purpose:** Historical execution records. Every time a test is executed, a row is written here. This is the table the classifier reads from and writes predictions back to.

**Row count:** 13,624

### Column Definitions

| Column | Type | Fill Rate | Description |
|---|---|---|---|
| `TC_ID` | Integer | 100% | Foreign key → TC_MASTER.TC_ID |
| `JIRA_ID` | String | 100% | Jira ticket ID |
| `EXEC_ID` | String | 100% | Execution run identifier (e.g. `UI_NTV_26_3`) |
| `INTRIM_STATUS` | String | 100% | **Status after engineer investigation — use this field, not FINAL_STATUS** |
| `FINAL_STATUS` | String | 100% | Raw execution result. **Do not use — gives a misleading picture** |
| `AUTO_FAILURE_REASON` | String | 45% | **The label to predict.** Filled by engineer after investigation. Empty = not yet classified |
| `USER_REMARKS` | String | 45% | Free-text written manually by engineer after investigation. Not suitable as classifier input |
| `FAILURE_REMARKS` | String | 0.4% | **Generated by the automation framework at time of failure.** The correct classifier input — very sparse in this CSV snapshot |
| `DEFECTID` | String | 14% | Jira defect ticket ID if a bug was raised |
| `MODULE` | String | 100% | Product module |
| `J_COMPONENT` | String | 100% | Component/feature area |
| `ASSIGNEE` | String | 100% | Engineer assigned to the execution |
| `BUILD_NO` | String | 0% | Build number — not populated in this snapshot |
| `EXEC_TIME` | String | 33% | Total execution time |
| `START_TIME` | String | 33% | Execution start timestamp |
| `END_TIME` | String | 33% | Execution end timestamp |

### INTRIM_STATUS Values

| Status | Count | Meaning |
|---|---|---|
| `PASSED` | 9,000 | Test passed. No failure to classify |
| `FAILED` | 1,631 | Test failed. Root cause needs classification |
| `BLOCKED` | 1,315 | Could not execute — usually an environment or config issue |
| `MAINTAINED` | 1,131 | Fix was applied, re-executed, and passed. Investigation complete |
| `MAINTAIN` | 475 | Direction/comment given but script still failing. Investigation in progress |
| `OOS` | 71 | Out of scope — usually retired due to an application change |
| `SCHEDULED` | 1 | Awaiting execution |

> **Rule:** Never use `FINAL_STATUS`. Always use `INTRIM_STATUS` as the status signal.

## Key Notes for the Classifier

### Primary Feature: FAILURE_REMARKS
- Contains machine-generated error text from the automation framework (e.g. *"expected [X] but found [Y]"*, *"Scenario grid could not be loaded"*)
- This is the correct classifier input — objective and system-generated, not manually written
- **Caveat:** Only 59 rows have this populated in the current CSV snapshot. The live database is expected to have significantly more

### FAILURE_REMARKS Coverage by Label (in this snapshot)

|---|---|
| APP-ISSUE | 36 |
| APP-CHANGE | 11 |
| SCRIPT-ISSUE | 6 |
| ENV-ISSUE | 6 |
| DATA-ISSUE | 0 |

### Why FAILURE_REMARKS Is Sparse — It Is the Live DB
This CSV **is** downloaded from the live database. FAILURE_REMARKS is not sparse due to a snapshot issue — it is genuinely sparse in production. The automation framework simply does not populate this field for most execution runs. What we see here is the real state of the data.

### The Critical Data Problem

| Metric | Count |
|---|---|
| Total rows in HIS_EXEC_REPORT | 13,624 |
| Rows WITH `FAILURE_REMARKS` populated | 59 (0.4%) |
| Labeled rows that also have `FAILURE_REMARKS` | 59 |
| The 24 unlabeled target rows with `FAILURE_REMARKS` | **0** |

**Every single one of the 24 rows we need to classify has an empty `FAILURE_REMARKS`.**

The `FAILURE_REMARKS` that do exist are all APP-ISSUE / APP-CHANGE / SCRIPT-ISSUE entries with messages like:
- *"estimate did not match!! expected [AUTO NATIONAL TV EST] but found [Select...]"*
- *"Scenario grid could not be loaded"*
- *"Buyer VPH is not as expected expected [2292.0] but found [5500.0]"*

These are framework-generated error strings — exactly the right kind of signal. But there are only 59 of them across the entire dataset.

### How We Arrived at the 24 Target Rows

| Step | Action | Remaining rows |
|---|---|---|
| Step 1 | Start with all rows in HIS_EXEC_REPORT | 13,624 |
| Step 2 | Remove rows already classified (have `AUTO_FAILURE_REASON`) | 7,520 |
| Step 3 | Remove `INTRIM_STATUS = PASSED` (test passed — nothing to classify) | 24 |
| **Step 4** | **What remains = rows that failed but have no classification yet** | **24** |

These 24 rows are failures (FAILED, BLOCKED, OOS, SCHEDULED) that engineers have not yet assigned a root-cause category to. **These are the rows the classifier must predict.**

### Revised Approach: What the Agent Will Actually Use

| Signal | Available on 24 target rows | Role |
|---|---|---|
| `FAILURE_REMARKS` | No (0/24) | Ideal input — not usable for this dataset |
| `INTRIM_STATUS` | Yes (24/24) | Primary rule-based signal |
| `USER_REMARKS` | Partial (some rows) | Secondary text signal when present |

**What the agent does:**
1. Uses `INTRIM_STATUS` as the primary rule signal (FAILED → APP-ISSUE, OOS → APP-CHANGE, etc.)
2. Uses `USER_REMARKS` text via semantic search against the 6,104 labeled rows when available
3. Where neither text signal exists, classifies by `INTRIM_STATUS` rule alone with a confidence note

### Fallback When FAILURE_REMARKS Is Empty

| INTRIM_STATUS | Predicted category | Historical accuracy |
|---|---|---|
| `FAILED` | APP-ISSUE | 99.5% |
| `BLOCKED` | ENV-ISSUE / Performance | 99.3% |
| `OOS` | APP-CHANGE | 81.7% |
| `MAINTAIN` | SCRIPT-ISSUE or DATA-ISSUE | Mixed — needs text |
| `MAINTAINED` | ENV-ISSUE | 94.7% |

## Classification Taxonomy (Full)

The target labels for `AUTO_FAILURE_REASON`, as defined by the team:

| Group | Label | Current DB value | Key signals |
|---|---|---|---|
| Product bug | App-Issue (Defect) | `APP-ISSUE` | Assertion errors, "expected X but found Y", `INTRIM_STATUS=FAILED` |
| Product bug | App-Change | `APP-CHANGE` | Jira change ticket in remarks (ADINFRA-XXXXXX), `OOS` status, "behavior changed" |
| Auto bug | Data Issue | `DATA-ISSUE` | Test data setup failures, CSV/data file keywords |
| Auto bug | Script Issue | `SCRIPT-ISSUE` | Java throwable/exception, `MAINTAIN` status, sync-related errors |
| System issue | Performance Issue | `ENV-ISSUE` *(combined)* | "passed on rerun", "passed locally", "pass from QB", intermittent failures |
| System issue | Sync Issue | `ENV-ISSUE` *(combined)* | "sync issue", timing failures, manually verified but script still failing |
| To investigate | Yet to be analyzed | *(unlabeled)* | Low confidence — valid output, not an error |

> **Note:** `ENV-ISSUE` in the current DB conflates **Performance** and **Sync** issues. The new classifier will distinguish between them. The 2,659 existing ENV-ISSUE rows will need a re-classification pass once the system is live.
